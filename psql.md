# `psql` Cheatsheet

`psql` is PostgreSQL's interactive terminal and command-line client. It sends
SQL to the server and handles backslash commands locally in the client.

Last checked: 2026-07-16. Examples target PostgreSQL 18. Older clients may not
support every option shown here.

## Quick start

Connect using local defaults:

```bash
psql
psql -d app
```

Connect over TCP:

```bash
psql \
  --host 'db.example.com' \
  --port '5432' \
  --username 'app' \
  --dbname 'app'
```

Connect through a Unix-domain socket directory:

```bash
psql --host '/var/run/postgresql' --username 'app' --dbname 'app'
```

Connect with a URI or keyword/value string:

```bash
psql 'postgresql://app@db.example.com:5432/app?sslmode=verify-full'

psql \
  "host=db.example.com port=5432 dbname=app user=app sslmode=verify-full"
```

Run one query and exit:

```bash
psql -X -d app -c 'SELECT current_database(), current_user;'
```

Run a script and stop on the first error:

```bash
psql \
  --no-psqlrc \
  --set=ON_ERROR_STOP=1 \
  --dbname=app \
  --file=script.sql
```

Inside `psql`:

```text
\conninfo   Show the current connection
\dt         List visible tables
\d table    Describe a table
\x auto     Use vertical output when results are wide
\timing on  Show elapsed time
\q          Quit
```

## Command-line options

Connection options:

| Short | Long | Purpose |
| --- | --- | --- |
| `-d` | `--dbname=DBNAME` | Database name, connection string, or URI |
| `-h` | `--host=HOST` | Server host or Unix socket directory |
| `-p` | `--port=PORT` | Server port |
| `-U` | `--username=USER` | Database role |
| `-w` | `--no-password` | Never prompt for a password |
| `-W` | `--password` | Prompt before the first connection attempt |
| `-l` | `--list` | List databases and exit |

Execution options:

| Short | Long | Purpose |
| --- | --- | --- |
| `-c` | `--command=COMMAND` | Run one SQL string or one backslash command |
| `-f` | `--file=FILE` | Read commands from a file |
| `-1` | `--single-transaction` | Wrap all `-c` and `-f` actions in one transaction |
| `-v` | `--set=NAME=VALUE` | Set a `psql` variable |
| `-X` | `--no-psqlrc` | Do not load system or user startup files |
| `-q` | `--quiet` | Suppress welcome and informational messages |
| `-n` | `--no-readline` | Disable line editing, completion, and history |

Output options:

| Option | Purpose |
| --- | --- |
| `--csv` | CSV output |
| `-A`, `--no-align` | Unaligned output |
| `-t`, `--tuples-only` | Hide headers and row-count footers |
| `-F SEP` | Set the unaligned field separator |
| `-R SEP` | Set the unaligned record separator |
| `-x`, `--expanded` | Expanded output |
| `-P NAME=VALUE` | Set a printing option |
| `-o FILE` | Send query output to a file |
| `-L FILE` | Duplicate query output to a log file |

Debugging and echo options:

| Option | Purpose |
| --- | --- |
| `-a`, `--echo-all` | Echo all nonempty script input lines |
| `-e`, `--echo-queries` | Echo SQL sent to the server |
| `-b`, `--echo-errors` | Echo failed SQL to standard error |
| `-E`, `--echo-hidden` | Show SQL generated by backslash commands |
| `-s`, `--single-step` | Confirm each command before sending it |

Help:

```bash
psql --help
psql --help=commands
psql --help=variables
psql --version
```

## Connection defaults

When values are omitted, the usual defaults are:

| Parameter | Default |
| --- | --- |
| Host | Local Unix-domain socket on Unix; `localhost` on Windows |
| Port | Compile-time default, usually `5432` |
| User | Current operating-system user |
| Database | Same name as the selected database user |

This common failure:

```text
database "alice" does not exist
```

often means both `-d` and `PGDATABASE` were omitted, so `psql` tried the
database named after the user. Specify a known maintenance database:

```bash
psql -d postgres
```

Frequently used libpq environment variables:

| Variable | Connection parameter |
| --- | --- |
| `PGHOST` | Host or socket directory |
| `PGPORT` | Port |
| `PGDATABASE` | Database |
| `PGUSER` | User |
| `PGAPPNAME` | Application name |
| `PGCONNECT_TIMEOUT` | Connection timeout in seconds |
| `PGSSLMODE` | TLS policy |
| `PGSSLROOTCERT` | Trusted CA file |
| `PGTARGETSESSIONATTRS` | Required server state, such as `read-write` |
| `PGSERVICE` | Connection service name |
| `PGSERVICEFILE` | Per-user service file |
| `PGPASSFILE` | Password file |
| `PGOPTIONS` | Server settings passed at connection time |

Example:

```bash
export PGHOST='db.example.com'
export PGPORT='5432'
export PGDATABASE='app'
export PGUSER='app'
export PGAPPNAME='manual-psql'
export PGCONNECT_TIMEOUT='5'

psql
```

Command-line and connection-string values override fallback environment
values. A service entry overrides environment values, and explicit values in
the same connection string override the service entry.

## Connection strings and TLS

Keyword/value form:

```bash
psql \
  "host=db.example.com port=5432 dbname=app user=app connect_timeout=5 sslmode=verify-full"
```

Quote the entire string in the shell. Inside the string, quote individual
values containing spaces:

```bash
psql "host=db.example.com dbname=app options='-c statement_timeout=5min'"
```

URI form:

```bash
psql 'postgresql://app@db.example.com:5432/app?connect_timeout=5&sslmode=verify-full'
```

Percent-encode characters that have special meaning in a URI. Avoid embedding
the password: URIs can leak through shell history, process listings, logs, and
error reports.

Try several hosts and require a writable server:

```bash
psql \
  'postgresql://app@db1.example.com:5432,db2.example.com:5432/app?target_session_attrs=read-write&connect_timeout=5&sslmode=verify-full'
```

Common `sslmode` values:

| Value | Behavior |
| --- | --- |
| `disable` | Do not use TLS |
| `prefer` | Try TLS first, then non-TLS; usually the default |
| `require` | Require encryption, but do not require identity verification |
| `verify-ca` | Require TLS and a certificate from a trusted CA |
| `verify-full` | Also verify that the certificate matches the host name |

For authenticated TLS, prefer `verify-full` and configure a trusted root
certificate. Confirm the negotiated connection with:

```text
\conninfo
```

## Passwords and connection services

Let `psql` prompt interactively when convenient. For unattended use, prefer a
password file over a command-line password or `PGPASSWORD`.

Unix password file:

```text
~/.pgpass
```

Format:

```text
hostname:port:database:username:password
```

Example:

```text
db.example.com:5432:app:app:replace-with-the-real-password
```

Secure it:

```bash
chmod 0600 ~/.pgpass
```

On Unix, libpq ignores a password file that is accessible by group or others.
The first matching line wins. The first four fields accept `*` wildcards;
place specific entries before broad ones. Escape `:` and `\` with `\`.

Use another password file without changing `HOME`:

```bash
PGPASSFILE='/secure/path/app.pgpass' psql 'service=app-prod'
```

Store reusable non-secret connection parameters in `~/.pg_service.conf`:

```ini
[app-prod]
host=db.example.com
port=5432
dbname=app
user=app
connect_timeout=5
sslmode=verify-full
```

Connect by service name:

```bash
psql 'service=app-prod'
psql 'service=app-prod application_name=maintenance'
```

Use `PGSERVICEFILE` to select a different per-user service file. Keep passwords
in the password file rather than the service file.

## Interactive basics

SQL can span several lines. A newline does not send a statement; a terminating
semicolon normally does:

```sql
SELECT id, created_at
FROM app.events
WHERE created_at >= now() - interval '1 day'
ORDER BY created_at DESC;
```

Backslash commands are processed by `psql` and do not end with semicolons:

```text
\dt
\d app.events
```

Frequently used controls:

| Input | Action |
| --- | --- |
| `;` | Finish and send the SQL statement |
| `\g` | Send the current query buffer |
| `\gx` | Send it with expanded output |
| `Ctrl+C` | Cancel a running query or clear current input |
| `Ctrl+D` | Exit at an empty prompt |
| `\q` | Quit |

The default prompt gives useful state hints:

| Prompt fragment | Meaning |
| --- | --- |
| `=>` | Ready; ordinary role |
| `=#` | Ready; database superuser |
| `->` | Statement is incomplete |
| `'>` or `">` | A quote is still open |
| `(>` | Parenthesis is still open |
| `=*>` | Inside a transaction |
| `=!>` | Inside a failed transaction; use `ROLLBACK` |

Exact prompts can vary with configuration.

## Help and connection information

`psql` help:

```text
\?                 Backslash-command help
\? commands        Backslash-command help
\? options         Command-line option help
\? variables       Special-variable help
```

SQL syntax help:

```text
\h
\h SELECT
\h ALTER TABLE
```

Connection and version information:

```text
\conninfo
\echo :DBNAME :USER :HOST :PORT
\echo :VERSION
\echo :SERVER_VERSION_NAME
```

From the shell:

```bash
psql --version
psql -X -d app -At -c 'SHOW server_version;'
```

Useful server-side identity query:

```sql
SELECT current_database() AS database,
       current_user AS effective_role,
       session_user AS session_role,
       inet_server_addr() AS server_address,
       inet_server_port() AS server_port,
       pg_backend_pid() AS backend_pid;
```

`inet_server_addr()` and `inet_server_port()` return `NULL` for a Unix-socket
connection.

Use a `psql` client at least as new as the newest server you administer.
Backslash commands are the most likely features to fail when an older client
connects to a newer server.

## The query buffer

The query buffer holds SQL typed but not yet sent.

| Command | Action |
| --- | --- |
| `\p` | Print the current buffer |
| `\r` | Clear the buffer |
| `\e` | Edit the buffer in an external editor |
| `\w FILE` | Write the buffer to a file |
| `\g` | Execute the buffer |
| `\gx` | Execute with expanded output |
| `\gdesc` | Show result column names and types without executing |

One-shot output formatting:

```text
SELECT id, payload FROM app.events
\g (expanded=on)

SELECT id, payload FROM app.events
\g (format=csv) /tmp/events.csv
```

If the buffer is empty, commands such as `\g`, `\gx`, `\gdesc`, `\gexec`, and
`\watch` operate on the most recently sent query. Check `\p` before using this
behavior with a query that changes data.

## Change connections

Reconnect using the current parameters:

```text
\c
```

Change only the database:

```text
\c analytics
```

Positional form:

```text
\c app app db.example.com 5432
```

Connection-string forms:

```text
\c service=app-prod
\c "host=db.example.com port=5432 dbname=app user=app sslmode=verify-full"
\c postgresql://app@db.example.com/app?sslmode=verify-full
```

Reuse unspecified parameters from the current connection while changing an
option:

```text
\c -reuse-previous=on "dbname=analytics sslmode=verify-full"
```

Positional `\c` reuses unspecified parameters by default. Conninfo and URI
forms do not unless `-reuse-previous=on` is given.

If an interactive reconnect fails, `psql` keeps the old connection. In a
non-interactive script, a failed `\connect` closes the old connection so the
script cannot silently continue against the wrong database.

## Inspect database objects

Databases, schemas, and roles:

| Command | Shows |
| --- | --- |
| `\l+` | Databases, sizes, tablespaces, and descriptions |
| `\dn+` | Schemas and privileges |
| `\du+` | Roles and attributes |
| `\drg` | Role memberships and grant options |
| `\drds` | Per-role and per-database settings |

Relations:

| Command | Shows |
| --- | --- |
| `\d` | Visible relations |
| `\d NAME` | Relation definition |
| `\d+ NAME` | Definition plus storage and other details |
| `\dt+ PATTERN` | Tables |
| `\di+ PATTERN` | Indexes |
| `\dv+ PATTERN` | Views |
| `\dm+ PATTERN` | Materialized views |
| `\ds+ PATTERN` | Sequences |
| `\dP+ PATTERN` | Partitioned tables and indexes |

Functions, types, and extensions:

| Command | Shows |
| --- | --- |
| `\df+ PATTERN` | Functions and procedures |
| `\sf+ FUNCTION` | Function or procedure definition |
| `\sv+ VIEW` | View definition |
| `\dT+ PATTERN` | Data types |
| `\dx+ PATTERN` | Installed extensions and their objects |

Privileges and configuration:

| Command | Shows |
| --- | --- |
| `\dp PATTERN` or `\z PATTERN` | Relation privileges |
| `\ddp` | Changed default privileges |
| `\dconfig` | Non-default server settings |
| `\dconfig *` | All server settings |
| `\db+` | Tablespaces |

Logical replication:

```text
\dRp+   Publications
\dRs+   Subscriptions
```

Examples:

```text
\dt public.*
\d+ public.events
\di+ public.events*
\df+ public.*
\dp public.*
```

## Name patterns and modifiers

Most `\d`-family commands accept a pattern:

| Pattern | Meaning |
| --- | --- |
| `*` | Any sequence of characters |
| `?` | Any one character |
| `schema.object` | Match schema and object separately |
| `"MixedCase"` | Preserve case and match special characters literally |

Examples:

```text
\dt public.*
\dt *.*
\df *calculate*
\d "CaseSensitiveTable"
\dtS+ pg_catalog.pg_*
```

Common command-name modifiers:

| Modifier | Meaning |
| --- | --- |
| `+` | More detail, often including size and description |
| `S` | Include system objects |
| `x` | Expanded display |

Without a pattern, many commands show only objects visible through the current
`search_path` and hide system objects. Use `\dt *.*` to check whether a table
exists in another schema.

## Output formatting

Useful interactive settings:

```text
\x auto
\pset null '(null)'
\pset border 2
\pset linestyle unicode
\pset format wrapped
\pset columns 120
```

Inspect all current printing settings:

```text
\pset
```

Main output modes:

| Mode | Use |
| --- | --- |
| `aligned` | Default human-readable table |
| `wrapped` | Human-readable output constrained to a width |
| `expanded` | One field per line, useful for wide rows |
| `unaligned` | Separator-delimited output without escaping |
| `csv` | Machine-readable CSV with quoting |
| `html`, `asciidoc`, `latex` | Tables for documents |

Toggle expanded output:

```text
\x
\x on
\x off
\x auto
```

CSV from the shell:

```bash
psql \
  --no-psqlrc \
  --csv \
  --dbname=app \
  --command='SELECT id, created_at FROM app.events ORDER BY id' \
  > events.csv
```

Tab-separated output in Bash:

```bash
psql \
  --no-psqlrc \
  --no-align \
  --tuples-only \
  --field-separator=$'\t' \
  --dbname=app \
  --command='SELECT id, created_at FROM app.events ORDER BY id'
```

Compact single-value output:

```bash
database_name="$(
  psql -X -A -t -d app -c 'SELECT current_database()'
)"
```

Unaligned output does not escape separators or newlines inside values. Use CSV
or `COPY ... FORMAT csv` when values are arbitrary.

## Output destinations and the pager

Send all subsequent query output to a file, then reset to standard output:

```text
\o /tmp/query-output.txt
SELECT * FROM app.events;
\d+ app.events
\o
```

One query only:

```text
SELECT * FROM app.events
\g /tmp/events.txt
```

Pipe query output to a local program:

```text
SELECT * FROM app.events
\g |gzip -c > /tmp/events.txt.gz
```

`\o` redirects query results, command tags, notices, and database-querying
backslash-command output, but not error messages. Use shell redirection when
standard error must also be captured.

Write script messages to the active query-output channel:

```text
\qecho 'starting report'
```

Pager control:

```text
\pset pager on
\pset pager off
\pset pager always
```

Select a pager from the shell:

```bash
export PSQL_PAGER='less -S'
```

An empty `PSQL_PAGER` disables the pager. `\watch` uses
`PSQL_WATCH_PAGER` separately.

## Timing and repeated queries

Show client-observed elapsed time for every SQL statement:

```text
\timing on
\timing off
```

Run a query repeatedly until interrupted:

```text
SELECT clock_timestamp(),
       count(*) AS active_sessions
FROM pg_stat_activity
WHERE state = 'active'
\watch 2
```

PostgreSQL 18 `psql` also supports named `\watch` options:

```text
SELECT id, status FROM app.jobs WHERE status = 'queued'
\watch interval=2 count=10 min_rows=1
```

This runs every two seconds, at most ten times, and stops if the query returns
fewer than one row. Press `Ctrl+C` to stop an unlimited watch.

For server execution detail, use SQL:

```sql
EXPLAIN (ANALYZE, BUFFERS, WAL, SETTINGS)
SELECT *
FROM app.events
WHERE id = 42;
```

`EXPLAIN ANALYZE` executes the statement. Use `EXPLAIN` without `ANALYZE` when
executing a data-changing statement would be unsafe.

## Variables and safe interpolation

Set, inspect, and remove `psql` variables:

```text
\set tenant_id 42
\echo :tenant_id
\set
\unset tenant_id
```

Pass values from the shell:

```bash
psql \
  -X \
  -v tenant_id='42' \
  -v schema_name='app' \
  -v table_name='events' \
  -d app \
  -f report.sql
```

Interpolation forms:

| Form | Result | Use |
| --- | --- | --- |
| `:name` | Raw text | Trusted SQL fragments only |
| `:'name'` | Quoted SQL literal | Data values |
| `:"name"` | Quoted SQL identifier | Schema, table, or column names |
| `:{?name}` | `TRUE` or `FALSE` | Test whether a variable exists |

Safe examples:

```sql
SELECT *
FROM :"schema_name".:"table_name"
WHERE tenant_id = :'tenant_id'::bigint;
```

Do not write:

```sql
SELECT * FROM :table_name WHERE tenant_id = :tenant_id;
```

Raw interpolation copies text directly into the SQL command. It is unsafe for
untrusted values and can change the command's structure.

Variable interpolation does not occur inside an SQL string or quoted
identifier:

```sql
-- Wrong: this is the literal text ":tenant_id"
SELECT ':tenant_id';

-- Correct:
SELECT :'tenant_id';
```

Bind values through the extended query protocol for the next execution:

```text
SELECT $1::integer + $2::integer
\bind 20 22
\g
```

`\bind` affects only the next execution. It is useful for testing parameterized
queries without text interpolation.

## Capture results with `\gset`

`\gset` requires exactly one result row. It creates one `psql` variable per
column:

```text
SELECT current_database() AS database,
       current_user AS role,
       pg_backend_pid() AS pid
\gset session_

\echo :session_database :session_role :session_pid
```

The optional prefix avoids collisions with existing variables. A `NULL` result
unsets the corresponding variable. If the query fails or does not return
exactly one row, no variables are changed.

Useful built-in state variables:

| Variable | Meaning |
| --- | --- |
| `ROW_COUNT` | Rows returned or affected by the last query |
| `SQLSTATE` | SQLSTATE from the last query; `00000` on success |
| `ERROR` | Whether the last query failed |
| `LAST_ERROR_SQLSTATE` | SQLSTATE of the most recent failed query |
| `LAST_ERROR_MESSAGE` | Primary message of the most recent failed query |
| `DBNAME`, `USER`, `HOST`, `PORT` | Current connection |
| `SERVER_VERSION_NUM` | Numeric server version |

Example:

```text
UPDATE app.jobs SET status = 'ready' WHERE status = 'new';
\echo changed :ROW_COUNT rows
```

## Conditional scripts

`\if` evaluates a Boolean value after variable and backquote expansion. It
does not evaluate arbitrary SQL or numeric expressions.

Have SQL compute the condition:

```sql
SELECT to_regclass('app.events') IS NOT NULL AS have_events
\gset

\if :have_events
  \echo 'app.events exists'
  SELECT count(*) FROM app.events;
\else
  \warn 'app.events does not exist'
\endif
```

Test whether a variable was supplied:

```text
\if :{?tenant_id}
  \echo 'tenant_id is set'
\else
  \warn 'tenant_id is not set'
\endif
```

Conditional blocks can be nested. Every `\if` through `\endif` in one block
must be in the same source file; a block cannot begin in one included file and
end in another.

## Run SQL from the shell

Run SQL:

```bash
psql -X -d app -c 'SELECT now();'
```

Run one backslash command:

```bash
psql -X -d app -c '\dt public.*'
```

Repeat `-c` and `-f` in the required order:

```bash
psql \
  -X \
  -d app \
  -c '\timing on' \
  -c 'SELECT count(*) FROM app.events' \
  -f cleanup.sql
```

One `-c` argument must be either a server-parsable SQL string or one complete
backslash command. Do not mix SQL and a backslash command in the same `-c`
argument.

Multiple SQL statements inside one `-c` string are sent as one server request
and are therefore executed as one transaction unless the string contains
explicit transaction control. Repeated `-c` arguments are separate requests.

Prefer `-f` over shell input redirection for scripts:

```bash
psql -X -d app -f migration.sql
```

`-f` gives `psql` the file name and produces better line-number diagnostics.

Inside `psql`:

```text
\i /absolute/path/script.sql
\ir sibling-script.sql
```

`\ir` resolves a relative path against the directory containing the current
script. `\i` resolves it against `psql`'s current working directory.

## Reliable automation

A strong default for a transactional script:

```bash
psql \
  --no-psqlrc \
  --no-password \
  --set=ON_ERROR_STOP=1 \
  --single-transaction \
  --dbname='service=app-prod' \
  --file=migration.sql
```

Why:

| Setting | Effect |
| --- | --- |
| `--no-psqlrc` | Prevent local startup customization from changing the run |
| `--no-password` | Fail instead of hanging on an unattended prompt |
| `ON_ERROR_STOP=1` | Stop at the first SQL or meta-command error |
| `--single-transaction` | Commit everything or roll back on error |
| `--file` | Include the file and line number in diagnostics |

`--single-transaction` works only with `-c` and `-f`. Do not use it when the
script contains explicit transaction control or commands that cannot run
inside a transaction block, such as `CREATE DATABASE`, `VACUUM`, or
`CREATE INDEX CONCURRENTLY`.

Useful script preamble:

```text
\set ON_ERROR_STOP on
\set ECHO errors
SET application_name = 'schema-migration';
SET lock_timeout = '5s';
SET statement_timeout = '5min';
```

Apply connection-time limits without changing the script:

```bash
PGOPTIONS='-c lock_timeout=5s -c statement_timeout=5min' \
  psql -X -w -v ON_ERROR_STOP=1 -d app -f maintenance.sql
```

Use `-q` only when reduced informational output is useful; do not hide errors.

## Exit status

`psql` uses these process exit codes:

| Code | Meaning |
| --- | --- |
| `0` | Normal completion |
| `1` | Fatal `psql` error, such as a missing file |
| `2` | Connection to the server failed during a non-interactive session |
| `3` | A script error stopped execution because `ON_ERROR_STOP` was set |

Always enable `ON_ERROR_STOP` when a shell script must reliably detect SQL
errors:

```bash
if ! psql -X -w -v ON_ERROR_STOP=1 -d app -f job.sql; then
  echo 'database job failed' >&2
  exit 1
fi
```

## Transactions

`psql` uses autocommit by default: every successful statement commits unless
you explicitly start a transaction.

```sql
BEGIN;

UPDATE app.accounts
SET enabled = false
WHERE id = 42;

-- Inspect the result before deciding.
SELECT id, enabled FROM app.accounts WHERE id = 42;

COMMIT;
-- or: ROLLBACK;
```

Recover from an error in an ordinary transaction:

```sql
ROLLBACK;
```

Use savepoints for manual partial recovery:

```sql
BEGIN;
SAVEPOINT before_change;

UPDATE app.accounts SET enabled = false WHERE id = 42;

ROLLBACK TO SAVEPOINT before_change;
COMMIT;
```

Client-side autocommit control:

```text
\set AUTOCOMMIT off
\set AUTOCOMMIT on
```

With autocommit off, `psql` issues an implicit `BEGIN` before eligible
statements. You must `COMMIT` successful work and `ROLLBACK` failed work. Work
left uncommitted when `psql` exits is lost.

For a friendlier interactive transaction workflow:

```text
\set ON_ERROR_ROLLBACK interactive
```

Inside a transaction, this creates an implicit savepoint before each statement
and rolls back only the failing statement. It adds round trips and is best kept
for interactive use rather than batch scripts.

Backslash commands run in the client. Their local side effects, such as
writing a file or running a shell command, are not rolled back with the SQL
transaction.

## Import and export with `\copy`

`\copy` reads and writes files on the `psql` client machine with the operating
system permissions of the client user.

Export a table:

```text
\copy app.events TO '/tmp/events.csv' WITH (FORMAT csv, HEADER true)
```

Export a query:

```text
\copy (SELECT id, created_at FROM app.events ORDER BY id) TO '/tmp/events.csv' WITH (FORMAT csv, HEADER true)
```

Import CSV:

```text
\copy app.events (id, created_at) FROM '/tmp/events.csv' WITH (FORMAT csv, HEADER true)
```

Inline data in a script:

```text
\copy app.labels (id, label) FROM STDIN WITH (FORMAT csv)
1,alpha
2,beta
\.
```

Compress or decompress with a local program:

```text
\copy (SELECT * FROM app.events) TO PROGRAM 'gzip -c > /tmp/events.csv.gz' WITH (FORMAT csv, HEADER true)
\copy app.events FROM PROGRAM 'gzip -dc /tmp/events.csv.gz' WITH (FORMAT csv, HEADER true)
```

`PROGRAM` runs a shell command as the local client user. Never construct it
from untrusted text.

`\copy` has special parsing rules:

- The whole command must be on one line.
- `psql` variable interpolation and backquote expansion are not performed.
- Data passes through the client/server connection.

For a multiline query or variable-aware output path, use server SQL
`COPY ... TO STDOUT` followed by `\g`:

```text
COPY (
  SELECT id, created_at
  FROM app.events
  ORDER BY id
) TO STDOUT WITH (FORMAT csv, HEADER true)
\g /tmp/events.csv
```

SQL `COPY ... TO '/server/path'` instead accesses the database server's file
system and requires server-side privileges. Do not confuse it with `\copy`.

## Generate and execute SQL

`\gexec` treats every non-NULL field returned by a query as SQL and executes
the generated statements in row order, left to right.

Preview generated SQL first:

```sql
SELECT format('ANALYZE %I.%I;', schemaname, tablename)
FROM pg_tables
WHERE schemaname = 'app'
ORDER BY tablename;
```

Then rerun the generator with `\gexec` instead of a semicolon:

```text
SELECT format('ANALYZE %I.%I;', schemaname, tablename)
FROM pg_tables
WHERE schemaname = 'app'
ORDER BY tablename
\gexec
```

Use `format('%I', value)` for identifiers and `format('%L', value)` for SQL
literals. Do not concatenate untrusted names or values.

Recommended safeguards:

```text
\set ON_ERROR_STOP on
\set ECHO queries
```

Generated fields must contain SQL, not backslash commands or `psql` variable
references. `\gexec` can make broad changes very quickly; verify the connection
and preview every generated statement.

## Shell, editor, and history

Local shell and directory commands:

```text
\! pwd
\! ls -la
\cd /tmp
```

These commands run on the client, not on the database server.

Editor commands:

| Command | Action |
| --- | --- |
| `\e` | Edit the current or previous query |
| `\ef FUNCTION` | Edit a function or procedure definition |
| `\ev VIEW` | Edit a view definition |
| `\w FILE` | Save the current or previous query |

Editor selection order is `PSQL_EDITOR`, `EDITOR`, then `VISUAL`:

```bash
export PSQL_EDITOR='vim'
```

History:

```text
\s
\s /tmp/psql-history.txt
\set HISTCONTROL ignoreboth
\set HISTFILE ~/.psql_history-:DBNAME
```

`ignoreboth` omits duplicate commands and commands beginning with a space.
History suppression is not a password-management strategy; use `\password`
when changing a role password:

```text
\password
\password app
```

`\password` prompts without placing the cleartext password in command history
or the server log.

## Session settings

Inspect and change server parameters for the current session:

```sql
SHOW search_path;
SHOW statement_timeout;

SET application_name = 'manual-maintenance';
SET statement_timeout = '5min';
SET lock_timeout = '5s';
SET search_path TO pg_catalog, app;

RESET lock_timeout;
RESET ALL;
```

Inspect settings through `psql`:

```text
\dconfig
\dconfig *timeout*
\dconfig *
```

Client encoding:

```text
\encoding
\encoding UTF8
```

Current session identity:

```sql
SELECT current_database(),
       current_user,
       session_user,
       current_schema(),
       pg_backend_pid();
```

Current transaction and recovery state:

```sql
SELECT txid_current_if_assigned(),
       pg_is_in_recovery();
```

Settings changed with `SET` normally last only for the current connection.

## Startup file

Unless `-X` is used, `psql` loads:

1. The system `psqlrc`.
2. The user's `~/.psqlrc`.

Find the system configuration directory:

```bash
pg_config --sysconfdir
```

Example `~/.psqlrc`:

```text
\set QUIET 1
\set COMP_KEYWORD_CASE upper
\set HISTCONTROL ignoreboth
\set HISTFILE ~/.psql_history-:DBNAME
\set ON_ERROR_ROLLBACK interactive
\pset null '(null)'
\pset linestyle unicode
\set PROMPT1 '%n@%m:%/%R%x%# '
\unset QUIET
```

Version-specific files are supported:

```text
~/.psqlrc-18
~/.psqlrc-18.4
```

The most specific matching file is preferred over the unversioned file.
`PSQLRC` can point to a different personal startup file.

Use `-X` in scripts, tests, and reproducible commands so personal formatting,
variables, SQL, and shell hooks do not affect the run.

## Security checklist

- Confirm `\conninfo`, `current_database()`, and `current_user` before a
  destructive command.
- Prefer `sslmode=verify-full` for authenticated TLS connections.
- Keep passwords out of command arguments, URIs, environment variables, SQL,
  service files, and history.
- Use `~/.pgpass` with mode `0600` or another protected password source.
- Use `\password` rather than typing `ALTER ROLE ... PASSWORD` interactively.
- Use `:'value'` and `:"identifier"` instead of raw `:value` interpolation.
- Treat `\!`, backquote expansion, output pipes, and `\copy ... PROGRAM` as
  local code execution.
- Treat `\gexec` as generated-code execution; preview its output.
- Use `-X` for automation and when the startup file is not trusted.
- Use `-w` in unattended jobs so a password prompt cannot hang the process.
- Set `ON_ERROR_STOP` in every script that must fail on SQL errors.
- Add `lock_timeout` and `statement_timeout` to maintenance automation.
- Avoid parsing aligned output; use CSV or a deliberate machine-readable mode.
- Clear or tightly control `search_path` when running administrative SQL in a
  database where untrusted users can create objects in searched schemas.

## Troubleshooting

### Wrong database or role

```text
\conninfo
\echo :DBNAME :USER :HOST :PORT
SELECT current_database(), current_user, session_user;
```

Remember that the default database name is the database user name.

### Local socket connection fails

```bash
pg_isready
pg_isready -h '/var/run/postgresql' -p '5432'

psql -h '/var/run/postgresql' -p '5432' -d postgres
```

Check that the server is running and that `-h` names the directory containing
the socket, not the socket file itself.

### TCP connection is refused or times out

```bash
pg_isready -h 'db.example.com' -p '5432' -t '5'
```

Check DNS, routing, firewalls, the server's `listen_addresses`, and port. If
the connection reaches the server but is rejected, inspect `pg_hba.conf` and
the authentication error separately.

### Peer authentication fails

Peer authentication checks the operating-system user. `-U` changes the
requested database role but not the operating-system identity. Use the intended
OS account or configure an appropriate authentication rule.

### Password authentication fails

Force a clean prompt to distinguish a stored-password problem:

```bash
psql -X -W -h 'db.example.com' -U 'app' -d 'app'
```

If prompting works, check:

- `~/.pgpass` permissions are `0600`.
- Host, port, database, and user match the first applicable password-file row.
- `PGPASSFILE` and `HOME` point where expected.
- The password entry escapes any `:` or `\`.

### Relation does not exist

```text
\conninfo
SHOW search_path;
\dn
\dt *.*
\d schema_name.table_name
```

Check the database, schema, case-sensitive quoted name, and `search_path`.

### Current transaction is aborted

```sql
ROLLBACK;
```

The default transaction behavior rejects further SQL after an error until the
failed transaction is rolled back.

### Output looks different in a script

Startup files and terminal detection affect formatting and the pager. Re-run
with explicit settings:

```bash
psql -X -A -t -P pager=off -d app -f script.sql
```

### A backslash command fails against a newer server

```text
\echo client=:VERSION_NAME server=:SERVER_VERSION_NAME
```

Install a `psql` client matching the server major version or a newer supported
client.
